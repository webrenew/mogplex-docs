Mogplex can run with either direct provider credentials or your Mogplex
account. On startup, it resolves auth in a fixed order.

## Resolution order

1. **Environment variable first.** If a provider env var is set in your shell,
   Mogplex uses it.

   | Provider   | Environment variable           |
   | ---------- | ------------------------------ |
   | Mogplex    | `MOGPLEX_API_KEY`              |
   | Anthropic  | `ANTHROPIC_API_KEY`            |
   | OpenAI     | `OPENAI_API_KEY`               |
   | Google     | `GOOGLE_GENERATIVE_AI_API_KEY` |
   | Groq       | `GROQ_API_KEY`                 |
   | Mistral    | `MISTRAL_API_KEY`              |
   | DeepSeek   | `DEEPSEEK_API_KEY`             |
   | xAI        | `XAI_API_KEY`                  |
   | Cohere     | `COHERE_API_KEY`               |
   | Vercel     | `VERCEL_API_TOKEN`             |

2. **Fallback to stored credentials.** If the env var is absent, Mogplex reads
   `~/.mogplex/auth.json`.

3. **No credentials means an interactive chooser or a clear failure.** Running
   bare `mogplex` in an interactive terminal opens the auth chooser. In
   non-interactive contexts, Mogplex exits with a clear message telling you to
   authenticate first.

The practical rule is simple: shell env vars outrank everything else.

## What account-backed login means

Browser login is not the same thing as storing a direct provider key locally.

When you sign in with your Mogplex account, the CLI stores a `mogplex`
credential in `~/.mogplex/auth.json` and can reuse hosted account state such
as:

- synced model catalog
- remote MCP server definitions
- the account-backed model access configured for your Mogplex user

This is the cleanest setup when you want the web app and CLI to share models,
MCP configuration, and account identity.

If the account login succeeds but prompts still fail, the hosted account
usually does not have usable model access yet. Add the needed provider key in
[Web Settings](/web/settings).

## Entry points

### `mogplex`

Running bare `mogplex` is the normal entrypoint:

```bash
mogplex
```

If no auth is available yet, Mogplex prompts you to either:

- sign in with Mogplex in the browser
- store a direct provider API key locally

### `mogplex login`

If you want to jump straight into setup, `mogplex login` opens the same
interactive auth flow:

```bash
mogplex login
```

You can inspect the stored auth file without opening the TUI:

```bash
mogplex login status
```

### `/login`

Inside a live session, `/login` is still the in-session path for switching or
storing credentials. `/login mogplex` goes to browser login; `/login
<provider>` asks for a provider key directly.

## Provider keys

Direct provider credentials still work and still take precedence over the
account login path. This is the preferred pattern for CI, ephemeral shells, and
environments where you do not want credentials written to disk.

```bash
export OPENAI_API_KEY=sk-...
mogplex
```

If both a stored credential and an env var exist for the same provider, the env
var wins.

## `/logout` and what it really changes

Use `/logout` inside a live session to remove a stored credential from
`~/.mogplex/auth.json`.

Three details matter:

- **Env vars still win.** If the matching env var is set, the next session will
  still authenticate through that env var.
- **Treat logout as affecting the next session.** If the current session already
  constructed a provider adapter, restart the session when you need immediate
  certainty that the old credential is gone.
- **Logging out of Mogplex clears shared hosted state.** Remote model catalog
  data and remote MCP server sync are cleared when the Mogplex account token is
  no longer available.

## Common outcomes and what they mean

- `mogplex login status` shows only `mogplex`: the CLI is in account-backed
  mode.
- `mogplex login status` shows a provider entry and the session uses that
  provider: direct provider auth is active.
- the CLI behaves differently than `login status` suggests: check env vars
  first, because they outrank stored auth.
- account login succeeds but prompts fail: the hosted account probably still
  lacks model access in [Web Settings](/web/settings).

## Logging and credential safety

Session logs at `~/.mogplex/logs/<session-id>.jsonl` do not contain:

- API keys
- OAuth tokens or PATs
- raw contents of `~/.mogplex/auth.json`

The runtime redacts those values before writing logs to disk.

When `MOGPLEX_DEBUG=1` is set, debug logs can still capture prompt content or
tool output. Treat debug logs with the same care as your source code.
