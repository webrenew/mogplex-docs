Settings is where account- and runtime-level configuration lives in the web app.

If something about your Mogplex account, credentials, model catalog, or shared
tooling looks wrong, this is the first place to inspect.

## Set these up first

For a new account, do these in order:

1. Connect GitHub.
2. Confirm GitHub App coverage for the correct personal account or org.
3. Add the provider key or gateway key that gives your hosted account model
   access.
4. Add shared Connections or MCP servers only after the account basics are
   working.
5. Create CLI access tokens only if you need CLI or script authentication as
   Mogplex itself.

## What lives here

- **Account** state for GitHub identity, GitHub App coverage, and optional
  Personal Vercel linking
- **Connections** for external REST APIs and MCP servers your agents can use
- **API Keys** for encrypted provider and AI Gateway keys
- **CLI Access Tokens** for browser-based CLI sign-in and script access
- **Preferences** for theme and global defaults
- **Models** for enabling and choosing the model catalog the product can use

## Account, GitHub, and App coverage

GitHub is the primary account identity for Mogplex.

The Account area also shows GitHub App coverage, installation metadata, and the
repos currently synced under each installation. This is the fastest way to
answer a very common setup question:

**Is this repo merely visible through OAuth, or is it fully covered for
triggered work?**

Personal Vercel is optional. It only matters when you want user-billed
sandboxes tied to your own Vercel project.

The top of Settings is also where Mogplex surfaces the next setup step. In
practice that usually means one of these:

- install or upgrade to the GitHub App
- sync repositories into Projects
- link Personal Vercel so a user-billed sandbox can target your own project

## Connections and MCP

Connections cover two categories:

- standard REST API integrations
- MCP servers, including quick-add presets and custom endpoints

The current quick-add MCP presets include Zapier, Notion, Supabase,
Browserbase, Sentry, Sanity CMS, and Linear.

From Settings you can:

- add a preset-backed MCP server
- register a fully custom MCP endpoint or REST connection
- choose auth mode such as bearer token, API key, OAuth, or basic auth
- test a connection after setup
- enable, disable, reconnect, or remove an existing connection

The same MCP definitions can be synced down to the CLI, so this is the cleanest
place to manage shared servers once instead of configuring every local machine
by hand.

## API Keys vs CLI Access Tokens

The hosted settings surface distinguishes between provider credentials and CLI
tokens:

- **API Keys** are encrypted provider keys for AI Gateway, OpenAI, and
  Anthropic, and the page can verify whether the stored key is valid
- **CLI Access Tokens** are personal access tokens for Mogplex itself, with a
  name, optional expiration, last-used metadata, and revoke controls

Use **API Keys** when the hosted product needs model access.
Use **CLI Access Tokens** when the Mogplex CLI or other Mogplex-aware tooling
needs to authenticate as you.

CLI tokens are shown once at creation time, then managed by prefix afterward.

## Preferences and models

Settings also owns the user-level defaults that affect both the hosted product
and CLI sync paths:

- theme preference
- default model selection
- enabled hosted model catalog under the Models section

If an agent cannot use the model you expect, check the Models section before you
edit the agent. Model availability and model choice are different concerns.

## Common failure modes

- **Triggers page is empty**: GitHub OAuth may be connected, but App coverage is
  missing for the repo owner.
- **The CLI signed in, but no synced models or MCP entries appear**: the stored
  Mogplex account is present, but the hosted account may still lack model access
  or shared connections.
- **A connection exists but agent calls still fail**: test the connection in
  Settings before debugging agent prompts.
- **A sandbox asks for Personal Vercel**: that setup is optional overall, but
  required for user-billed sandbox flows.

## Why Settings matters

The web app and CLI share parts of this control plane.

Settings is the place to wire the hosted side once, then reuse those choices
from local workflows when the CLI syncs model catalogs, tokens, or MCP server
definitions.
