In the Mogplex app, the top-bar label is **Projects** and the route is
`/spaces`.

This is the operational entrypoint for repository-backed work. If you are asking
"which repo should I work on, is it ready, and how do I open it into a live
workspace?" this is the page you want.

## Terms that matter

- A **repo** is the imported GitHub repository record Mogplex knows about.
- A **project** is the grouping layer used to organize repos intentionally.
- A **workspace** is the active working session you open for a repo.
- A **sandbox** is the preview or runtime environment tied to that repo work.

## Use Projects when you need to

- sync or inspect imported repositories
- group repos into projects
- search, filter, favorite, or hide repos
- launch or stop preview sandboxes
- open a workspace session for a repo
- configure repo-specific settings such as root directory, dev command, env
  sync mode, and linked Vercel state
- create repo-scoped assignments or cron-style work from the repo surface

## Prerequisites

Projects works best after three setup steps are done:

1. GitHub identity is connected in [Settings](/web/settings).
2. The Mogplex GitHub App is installed for the user or org that owns the repos.
3. Repo sync has run at least once.

If you can see repos but cannot use webhook-backed features yet, that usually
means OAuth is connected but GitHub App coverage is still missing.

## First successful workflow

1. Sync the target repos into Mogplex.
2. Find the repo in Projects and confirm it appears under the right owner or
   project group.
3. Open the repo.
4. If no live preview exists yet, launch a sandbox.
5. Once the sandbox is ready, open a workspace session.
6. From that repo surface, decide whether the next step is:
   - local work in the workspace
   - standing routing in [Assignments](/web/assignments)
   - direct event routing in [Triggers](/web/triggers)
   - a richer workflow in [Automations](/web/automations)

## What the page is managing for you

- imported repository coverage and visibility
- project-level grouping
- repo-level operational settings
- sandbox launch and cleanup
- workspace entry
- monorepo sub-project entrypoints
- quick jumps into repo-bound agent work

Projects is where discovery turns into execution. It is not just a catalog.
Opening a repo can launch a sandbox, bind a workspace session to it, and expose
repo-specific actions that continue into the rest of the product.

## Operational signals to watch

- A repo is visible but not fully usable: GitHub OAuth exists, but GitHub App
  coverage is missing for that owner or org.
- A repo cannot launch a user-billed sandbox: Personal Vercel is either not
  linked or not bound for that repo's billing setup.
- A repo opens slowly: the sandbox probably needs to be launched first.
- A monorepo needs multiple entrypoints: use sub-project repo entries rather
  than pretending one root fits every target.

## If Projects is empty

The next step is usually one of:

- install or finish the GitHub App so repo sync can run cleanly
- open [Settings](/web/settings) and connect GitHub first
- link Personal Vercel if you specifically want user-billed sandboxes
