Observability is the runtime view for Mogplex work that has already started.

The full page is designed to answer two operator questions quickly:

1. what is happening right now across CLI, cloud, and automation
2. where do I go next to debug the specific bad thing

## What the page shows

- **Summary** for total calls, tokens, cost, and top-level health signals
- **Activity** for model calls across CLI, cloud, and automation
- **Runs** for job-level execution across triggers, assignments, and flows
- **Pressure** for suppressed, deferred, and failed dispatch starts
- **Automation Failures** for timeout, auth, provider, and configuration
  failures
- **Sandbox-linked drilldown** from call rows when execution touched a sandbox

This is the page where the product converges after something begins running.

## The debugging flow Mogplex expects

Start broad and narrow only when the signal demands it:

1. Look at **Summary** to confirm whether this is isolated or systemic.
2. Use **Runs** when you care about a job or workflow outcome.
3. Use **Pressure** when work is not starting or is being intentionally held.
4. Use **Automation Failures** when the job did start but the system itself
   failed around it.
5. Use **Activity** when you need the model-level detail: tokens, cost, tool
   usage, latency, or the exact call record.

## Filters and drilldown

- Filter by surface, status, repo, sandbox, or date range.
- Expand a call row to inspect the stored prompt/response context, tool calls,
  and sandbox metadata.
- Open sandbox health directly from a call when sandbox context is available.
- Repair, requeue, or cancel supported jobs from the runs surface.

The date-range presets currently include `1h`, `today`, `7d`, and `30d`.

## Practical workflows

### A trigger or assignment did not run

1. Check **Runs** for the expected repo, source, and agent.
2. If there is no run, check **Pressure** for suppression or deferral.
3. If the route still looks idle, go back to the source surface:
   [Triggers](/web/triggers), [Assignments](/web/assignments), or
   [Automations](/web/automations).

### A run happened but the result looked wrong

1. Find the job in **Runs**.
2. Expand into its latest call trail through **Activity**.
3. Inspect model choice, tokens, tool calls, and linked repo or sandbox
   context.

### Sandbox work looks unhealthy

1. Find the relevant call in **Activity**.
2. Open the sandbox-linked detail.
3. Jump into sandbox health instead of debugging the AI call in isolation.

## Read the signals correctly

- **Pressure** is about dispatch and scheduling behavior.
- **Automation Failures** is about concrete system failures such as timeouts,
  auth issues, and provider errors.
- **Activity** is about the model work itself.
- **Runs** is the best answer to "did the job happen, and what was its state?"

Observability is where [Triggers](/web/triggers), [Assignments](/web/assignments),
and [Automations](/web/automations) converge once something is already running.
