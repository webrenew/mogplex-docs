Mogplex has two surfaces that are meant to work together:

- the **web app** is the shared control plane for GitHub coverage, repos,
  agents, routing, settings, and observability
- the **CLI** is the local runtime for repo-in-place work from your terminal

You do not need to learn the whole product at once. The first useful question
is simpler:

**Where does this piece of work start?**

## Pick the right starting surface

- Start in **Web App** when you need account setup, GitHub coverage, repo sync,
  reusable agents, installation-backed routing, or shared observability.
- Start in **CLI** when you already have a local repo and want Mogplex to
  inspect, review, edit, or automate work from the terminal.
- Use **both** when the account and routing state should live in the web app,
  but the day-to-day work should happen close to the repo in your shell.

## Fast decision guide

- "I need Mogplex to see my repos and respond to GitHub events."
  Start with [Web App](/web), especially [Settings](/web/settings) and
  [Installations](/web/installations).
- "I already have the repo checked out and want Mogplex to do one task."
  Start with [CLI](/cli), then use `mogplex` or `mogplex exec`.
- "I need one GitHub event to wake one agent."
  Start with [Triggers](/web/triggers).
- "I need repo-bound standing work like PR review, issue triage, or cron."
  Start with [Assignments](/web/assignments) or [Projects](/web/spaces).
- "I need branching, delays, multiple agents, or a publishable workflow."
  Start with [Automations](/web/automations).

## Shortest first-day path

If you want one reliable setup order that works for most users, use this:

1. Open the [web app](/web) and connect GitHub in [Settings](/web/settings).
2. Confirm GitHub App coverage in [Installations](/web/installations), then
   sync repos in [Projects](/web/spaces).
3. Inspect or create the reusable agents you want in [Agents](/web/agents).
4. Choose the smallest routing surface that fits the job:
   - [Triggers](/web/triggers) for `one event -> one agent`
   - [Assignments](/web/assignments) for repo-bound standing work
   - [Automations](/web/automations) for multi-step workflows
5. Install the [CLI](/cli) and run a first local task in the repo.
6. Use [Observability](/web/observability) once work is actually running.

## Fastest end-to-end proof

If you only want to prove Mogplex is working before you invest in deeper setup,
install the CLI and run one session:

```bash
curl -fsSL https://www.mogplex.com/install.sh | sh
mogplex
```

If you are not authenticated yet, the CLI walks you into browser sign-in or
local provider-key setup.

Success looks like:

- `mogplex` opens an interactive session
- you can authenticate with either your Mogplex account or a provider key
- `mogplex exec "summarize this repository"` returns a normal one-off result
- the same account later appears correctly in [Web Settings](/web/settings) if
  you use account-backed login

If you prefer shell-managed credentials, direct provider env vars still work:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
# or
export OPENAI_API_KEY=sk-...
mogplex
```

## What good looks like

By the end of your first pass, you should be able to say yes to these:

- Mogplex can see the repos you expect.
- The repo owner has GitHub App coverage, not just OAuth visibility.
- At least one reusable agent is configured the way you expect.
- One local CLI task completes successfully.
- You know whether this job belongs in Triggers, Assignments, or Automations.
